# HF 0.4.31 — `validate` WCAG contrast audit produces `null:1` for tall compositions

## Summary

`hyperframes validate` reports WCAG contrast warnings with `ratio: null` and `fg: rgb(NaN,NaN,NaN), bg: rgb(undefined,undefined,undefined)` for any text element whose `getBoundingClientRect()` extends below the headless viewport's 1080-px height.

For a 1440×2560 vertical composition (typical for short-form mobile video), every text element below the first 1080 px hits this. The contrast audit cannot produce a valid measurement, so users see persistent contrast warnings that no CSS change can fix.

## Reproduction

1. Author a composition with `data-width="1440" data-height="2560"`.
2. Place a text element (e.g. a frosted-glass card with white text on dark fill) at viewport-Y 1280 (centre of 2560).
3. Run `hyperframes validate <project> --json`.

Expected: `"contrastFailures": 0` (white-on-dark passes >15:1 trivially).

Actual:
```json
{
  "time": 26.35,
  "selector": "#smoke-plate",
  "ratio": null,
  "wcagAA": false,
  "fg": "rgb(NaN,NaN,NaN)",
  "bg": "rgb(undefined,undefined,undefined)"
}
```

## Root cause

`src/commands/validate.ts` sets the puppeteer viewport unconditionally:

```ts
await page.setViewport({ width: 1920, height: 1080 });
```

The screenshot for the contrast audit is taken at viewport size (1920×1080). DOM elements at `y >= 1080` produce `rect.y` exceeding the screenshot canvas height.

In `src/commands/contrast-audit.browser.js`, the ring-sampling bounds clamp `x0`/`y0` only against `0`, not against `w-1`/`h-1`:

```js
var x0 = Math.max(0, Math.floor(rect.x) - 4);
var y0 = Math.max(0, Math.floor(rect.y) - 4);
```

So for `rect.y = 1280, h = 1080`, `y0 = 1276`, then `sample(x, y0=1276)` reads `px[(1276 * w + x) * 4]` — far past the end of `Uint8ClampedArray`, returning `undefined`. The arrays of pixel samples (`rr`, `gg`, `bb`) end up populated with `undefined` values that pass the `if (rr.length === 0)` guard, and `median(rr)` returns `undefined`. The composite math `Math.round(0 + undefined * 0)` → `NaN` propagates into the output.

## Suggested fix (one of)

### Minimal — `contrast-audit.browser.js`

Skip elements outside the screenshot, and double-clamp the ring bounds:

```js
if (rect.x + rect.width <= 0 || rect.x >= w || rect.y + rect.height <= 0 || rect.y >= h) continue;
var x0 = Math.max(0, Math.min(w - 1, Math.floor(rect.x) - 4));
var x1 = Math.max(0, Math.min(w - 1, Math.ceil(rect.x + rect.width) + 4));
var y0 = Math.max(0, Math.min(h - 1, Math.floor(rect.y) - 4));
var y1 = Math.max(0, Math.min(h - 1, Math.ceil(rect.y + rect.height) + 4));
```

This makes the symptom stop. Tall compositions still get partial coverage (only elements above 1080 px get audited).

### Proper — `validate.ts`

Set the viewport to the composition's `data-width`/`data-height`:

```ts
const root = await page.evaluate(() => {
  const el = document.querySelector("[data-composition-id][data-duration]");
  return el ? { w: parseInt(el.getAttribute("data-width") ?? "1920"), h: parseInt(el.getAttribute("data-height") ?? "1080") } : { w: 1920, h: 1080 };
});
await page.setViewport({ width: root.w, height: root.h });
```

(Plus the minimal fix above as belt-and-suspenders.) This audits the entire composition, not just the first 1080 px.

`patch-package` applies the minimal fix locally — see `docs/hyperframes-upgrade.md`. Patch file at `tools/compositor/patches/hyperframes+0.4.31.patch`.
